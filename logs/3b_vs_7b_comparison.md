# 3B vs 7B 한국어 쿼리 문제 비교 분석

- **분석일**: 2026-05-30
- **실험**: filesystem / github / memory 서버, EN/KO 쿼리 쌍

---

## 문제 잔존 여부 비교

| 문제 | 설명 | 3B | 7B | 원인 분류 |
|------|------|----|----|-----------|
| **P1** | 언어 전환 (한국어→영어/중국어) | 빈번 | **2건 잔존** | ✅ 한국어 고유 문제 |
| **P2** | 한국어 내용 번역·손실 | 전건 | ✅ 해소 | 모델 크기 문제 |
| **P3** | JSON 생성 오류 (outer `}` 누락) | 전건 | ✅ 해소 | 모델 크기 문제 |
| **P4** | 에이전트 오동작 (저장소 생성) | 1건 | ✅ 해소 | 모델 크기 문제 |
| **P5** | 도구 인수 추론 실패 | 전건 | ⚠️ 형태 변화 | MCP 서버 API 설계 문제 |
| **P6** | 컨텍스트 폭증 | 1건 | ✅ 해소 | 모델 크기 문제 |

---

## 각 문제 상세

### P1 — 언어 전환 (7B에서도 잔존, 한국어 고유 문제 확인)

7B도 한국어 쿼리에 영어/중국어로 응답하는 사례 발생:

```
쿼리:  "현재 디렉토리에 있는 모든 파일과 폴더 목록을 알려줘."
7B 응답: "当前目录下的文件和文件夹列表如下：..."  ← 중국어!
```

```
쿼리:  "메모리에서 MCP와 Qwen 정보를 불러와서 두 개의 관계를 한국어로 설명해줘."
7B 응답: "MCP is a protocol that connects AI models..."  ← 영어
```

- **결론**: 한국어 쿼리를 한국어로 처리하지 못하는 것은 Qwen 모델 계열의 고유 특성
- Qwen은 중국어 학습 데이터가 많아 한국어를 중국어로 혼동하는 경향
- 모델 크기(3B→7B)로 해소되지 않음 → **한국어 특화 모델 필요 (예: EXAONE, HyperCLOVA X)**

### P2 — 내용 번역·손실 (7B에서 해소)

```
3B: "MCP" 저장 시 관찰값 → "MCP is a protocol that connects AI models..." (영어 번역)
7B: "MCP" 저장 시 관찰값 → "Connecting AI models to external tools" (영어이지만 간결하게 저장)
```

- 7B도 영어로 저장하지만, 내용을 정확하게 반영하고 짧게 유지
- 3B처럼 과도한 번역·확장 없음 → JSON 길이 임계값 미초과

### P3 — JSON 생성 오류 (7B에서 완전 해소)

```
3B: 마지막 토큰 ']}\n' → outer } 누락 → 파싱 실패
7B: 마지막 토큰 '}}\n' → 올바른 JSON 생성
```

- 3B의 훈련 아티팩트(token id=23439)가 7B에서는 존재하지 않음

### P4 — 에이전트 오동작 (7B에서 해소)

```
3B: 커밋 조회 실패 → create_repository 호출 → 실제 저장소 생성
7B: 커밋 조회 성공 (sha='main' 직접 사용) → 저장소 생성 없음
```

- 7B의 더 나은 API 파라미터 추론 능력

### P5 — 도구 인수 추론 실패 (형태 변화, 부분 잔존)

```
3B: read_text_file(head=20) 반복 → 4회 실패 후 포기
7B: read_text_file(head=None, tail=None) → read_text_file(head=100, tail=None)
    → read_text_file(head=100, tail=100) → "head와 tail 동시 불가" 오류
    → 결국 max steps 도달, "파일 위치를 알려주시면..." 식의 hallucination
```

- **실제 원인**: `read_text_file` 도구의 API 설계 문제 — head와 tail이 둘 다 필수이지만 동시 사용 불가
- 모델 크기와 관계없이 이 API를 올바르게 호출하기 어려움
- **한국어 고유 문제 아님**: EN 쿼리도 동일하게 실패

### P6 — 컨텍스트 폭증 (7B에서 해소)

```
3B: github 커밋 조회 Step4에서 44,842 tokens
7B: 최대 64,794 tokens (github 파일목록)이지만 에이전트가 제때 종료
```

- 7B는 필요 정보를 빠르게 찾아 더 적은 스텝으로 종료

## 최종 결론: 한국어 쿼리로 인한 문제 vs 모델 크기 문제

```
한국어 고유 문제 (모델 크기로 해소 안 됨):
  P1. 언어 전환 — 한국어 쿼리에 영어·중국어 응답

모델 크기 문제 (7B에서 해소됨):
  P2. 내용 번역·손실
  P3. JSON 생성 오류
  P4. 에이전트 오동작
  P6. 컨텍스트 폭증

MCP 서버 API 설계 문제 (모델 무관):
  P5. 도구 인수 추론 실패 (head/tail 동시 필수이지만 동시 사용 불가)
```

**즉, 한국어 쿼리로만 발생하는 진짜 문제는 P1(언어 전환) 하나다.**

나머지는 소형 모델의 한계(P2~P4, P6)이거나 MCP 서버 설계 결함(P5)이며,
한국어가 이 문제들을 더 빈번하게 트리거하지만 근본 원인은 다르다.

## 한국어 MCP 실용화를 위한 권고

1. **언어 전환 방지**: Qwen 대신 한국어 특화 모델 사용 (EXAONE 3.0, HyperCLOVA X)
2. **현재 Qwen으로 최소화**: 시스템 프롬프트에 `"응답은 반드시 한국어로"` 명시
3. **MCP 서버 선택**: filesystem의 `read_text_file`처럼 API 설계가 복잡한 도구는 피하고, `list_directory`처럼 단순한 도구 우선 사용
4. **모델 크기**: Qwen2.5-7B 이상에서 P2~P4는 해소되므로, 7B+ 사용 권장
