## 발견된 문제점 (심각도 순)

### 🔴 P1 — JSON 내 한국어 인코딩 오류 (memory 서버)

**증상**
```
Expecting ',' delimiter: line 1 column 261 (char 260)
decoding failed on that specific part: 'e Model입니'
```
모델이 `create_entities` 도구를 호출할 때 인수 JSON 안에 한국어를 포함하면
멀티바이트 경계에서 JSON 파싱이 실패함.

**원인**
3B 소형 모델의 토크나이저가 JSON 직렬화 시 한국어 유니코드 문자를
토큰 경계에서 잘라내어 불완전한 JSON을 생성.

**영향**: memory 서버는 한국어 데이터 저장 자체가 불가능.

---

### 🔴 P1 — 도구 인수 오해 및 반복 실패 (fetch 서버)

**증상**
```
Argument signal has type 'null' but should be 'string'
→ 동일한 오류 4회 반복 → max steps 도달
```
모델이 `fetch` 도구의 선택적 인수 `signal`에 `null`을 넣었으나
서버가 string 타입을 요구. 오류 메시지를 받고도 같은 인수로 재호출.

**원인**: 소형 모델의 오류 회복(error recovery) 능력 부족.
오류를 인식하지만 수정 방향을 추론하지 못하고 반복.

---

### 🟠 P2 — 오류 시 언어 전환 (filesystem, fetch, github)

**증상**
한국어로 쿼리했으나 도구 호출 실패가 반복되면 영어로 응답 전환:
```
"Given the persistent issues with the tail/head arguments,
let's approach the task differently..."
```

**원인**: Qwen2.5-3B의 instruction-following 능력의 한계.
오류 복구 시나리오에서는 학습 데이터의 영어 비중이 높아
한국어 지시를 유지하지 못함.

---

### 🟠 P2 — 필수 도구 인수 파악 실패 (filesystem 서버)

**증상**
```
Calling tool: 'read_text_file' with arguments: {'path': 'README.md'}
→ "Argument tail is required"
→ 동일 오류 4회 반복
```
`read_text_file`이 `head`와 `tail` 두 인수를 동시에 요구하는 구조인데,
모델이 이를 파악하지 못하고 `head`만 추가한 채 반복.

---

### 🟡 P3 — 컨텍스트 폭증으로 인한 응답 지연 (github 서버)

**증상**
- Step 1: 9,454 tokens → Step 4: 44,842 tokens
- Step 4 단독 소요 시간: **165초**

GitHub API 응답이 JSON 전체(SHA, URL 등 메타데이터 포함)로 반환되어
컨텍스트가 급격히 커짐. CPU 환경에서는 사실상 실용 불가 수준.

---

### 🟡 P3 — MCP 서버 초기 연결 타임아웃 (fetch 서버)

**증상**: 첫 번째 쿼리에서 30초 타임아웃 후 연결 실패.
두 번째 쿼리에서는 서버가 정상 기동됨.

**원인**: `mcp-fetch` npm 패키지 최초 실행 시 초기화 지연.

## 개선 방향

1. **모델 교체**: 3B → 7B 이상 모델 (또는 GPU 환경) 사용 시
   JSON 생성 품질과 오류 회복 능력이 크게 향상 예상
2. **structured_output=True 적용**: smolagents 1.25 경고 대응,
   도구 인수 생성 방식을 구조화하면 JSON 오류 감소 가능
3. **시스템 프롬프트 강화**: "항상 한국어로 응답하라" 지시를
   system prompt에 명시적으로 추가
4. **도구 스키마 단순화**: 복잡한 선택적 인수(signal 등)를 제거하거나
   default 값을 서버 측에서 처리하도록 수정
5. **메모리 서버 우회**: 한국어 데이터는 ASCII로 변환하거나
   영어 키워드로 저장 후 한국어 매핑 테이블 별도 관리
