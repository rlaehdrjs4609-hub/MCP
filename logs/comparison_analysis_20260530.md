# 영어 vs 한국어 대조 실험 분석 리포트

- **실험일**: 2026-05-30
- **모델**: Qwen/Qwen2.5-3B-Instruct (GPU: RTX A6000, bfloat16)
- **대상 서버**: filesystem, github, memory
- **목적**: 문제 원인이 "소형 모델 한계"인지 "한국어 처리 문제"인지 분리

---

## 정량 결과

| 서버 | EN 성공 | KO 성공 | EN 평균속도 | KO 평균속도 | KO 언어유지 |
|------|---------|---------|------------|------------|------------|
| filesystem | 2/2 | 2/2 | 8.1초 | 5.5초 | 1/2 |
| github | 2/2 | 2/2 | 22.5초 | 24.5초 | 1/2 |
| memory | 2/2 | 2/2 | 12.7초 | 15.8초 | 1/2 |

> **성공률 기준**: 오류 없이 최종 응답 반환 (응답 품질 무관)

---

## 핵심 발견

### 발견 1 — 성공률은 언어와 무관 (모델 능력 한계가 주 원인)

영어/한국어 모두 동일 쿼리에서 같은 방식으로 성공하거나 실패.

**filesystem — README 요약 (EN/KO 모두 실패)**
```
EN: read_text_file (tail 누락) → 5회 반복 → 영어 할루시네이션 응답
KO: read_text_file (tail 누락) → 5회 반복 → 영어 할루시네이션 응답
```
→ tail 인수를 못 찾는 것은 언어와 무관한 3B 모델의 추론 한계

**github — 커밋 조회 (EN/KO 모두 잘못된 sha 사용)**
```
EN: sha='HEAD' → 실패, sha='master' → 실패, 다른 도구 시도
KO: sha='latest' → 실패, search_repositories → 실패, sha='main' 로 자체 회복 성공
```
→ 잘못된 sha 값을 넣는 것은 언어와 무관. KO가 오히려 EN보다 회복력이 좋은 경우도 있음

---

### 발견 2 — JSON 내 한국어 인코딩 오류는 한국어 특이 문제 ✅

**memory — 정보 저장**
```
EN 쿼리: create_entities(observations=["Connecting AI models to external tools"])
         → 1스텝 성공 ✅ (1.6초)

KO 쿼리: create_entities(observations=["AI 모델과 외부 도구를 연결하는 프로토콜입니다."])
         → JSON 파싱 오류 5회 반복 ❌
         decoding failed: 'e Model입니', '도구를 연결하는', 'ations": '
```

**원인 분석**: 3B 모델의 토크나이저가 `<tool_call>` JSON을 생성할 때 한국어 멀티바이트 문자(UTF-8 3바이트)를 토큰 경계로 잘라 유효하지 않은 JSON 생성. 영어는 ASCII 1바이트라 이 문제가 발생하지 않음.

이 문제는 **언어 특이적**이며 모델 크기와 관계없이 소형 모델에서 발생 가능.

---

### 발견 3 — 오류 시 영어 전환은 언어 무관 패턴

EN/KO 모두 오류가 반복되면 영어로 fallback. KO 쿼리만의 문제가 아님.
단, KO 입력에서 영어 전환이 더 두드러지게 나타남 (언어 유지율 1/2).

---

### 발견 4 — 속도는 언어 간 차이 없음

filesystem에서 KO(5.5초)가 EN(8.1초)보다 오히려 빠름.
GPU 환경에서 한국어/영어 토큰 처리 속도 차이는 무시 가능.

---

## 결론: 원인 분류

| 문제 | 원인 | 한국어 특이? |
|------|------|------------|
| tail/head 인수 추론 실패 | 3B 모델 능력 한계 | ❌ 아님 |
| sha 값 오류 (HEAD/latest) | 3B 모델 능력 한계 | ❌ 아님 |
| JSON 내 한국어 파싱 오류 | 멀티바이트 토큰 경계 문제 | ✅ 한국어 특이 |
| 오류 시 영어 전환 | 모델 fallback 패턴 | 부분적 (KO에서 더 빈번) |
| 컨텍스트 팽창 (GitHub) | GitHub API 응답 크기 | ❌ 아님 |

**결론**: 대부분의 실패는 3B 소형 모델의 일반적 한계(도구 호출 오류 회복 불능)에 기인.
단, **JSON 내 한국어 인코딩 오류**는 순수 한국어 특이 버그로 확인됨.

---

## 개선 방향

1. **JSON 인코딩 문제 해결**: `structured_output=True` 적용 (smolagents 1.25 기본값 변경 예정)
   → 도구 인수를 별도 구조체로 생성해 한국어 멀티바이트 경계 문제 우회 가능

2. **모델 업그레이드**: 7B 이상 모델 사용 시 오류 회복 능력 향상 예상
   → RTX A6000 49GB VRAM 기준 7B bfloat16(~14GB) 충분히 수용 가능

3. **시스템 프롬프트 강화**: "항상 한국어로 응답하라" 명시 → 언어 전환 방지
