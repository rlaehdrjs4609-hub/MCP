## 문제 1 — 한국어 → 영어 언어 전환

**증상**: 한국어로 쿼리해도 도구 호출 실패가 반복되면 영어로 응답 전환.

```
쿼리:  "README.md 파일을 읽고 내용을 한국어로 요약해줘"
응답:  "It seems there is a recurring issue with the required 'tail' argument..."
```

**발생 패턴**: 도구 오류 2회 이상 반복 시 한국어 지시를 잊고 영어 fallback.

**대조 실험 결과**: 영어 쿼리도 오류 반복 시 동일하게 영어로 응답. 따라서 한국어 특이 문제는 아니지만, **한국어 쿼리는 영어에 비해 전환 빈도가 더 높음** (언어 유지율 1/2).

---

## 문제 2 — 한국어 내용 번역·손실

**증상**: 한국어 관찰값을 memory 서버에 저장할 때 모델이 임의로 영어로 번역.

```
입력:  "MCP는 AI 모델과 외부 도구를 연결하는 프로토콜이야"
저장됨: "MCP is a protocol that connects AI models and external tools."  ← 영어
```

**원인**: Qwen2.5-3B의 학습 데이터 편향 — tool call의 `observations` 필드에 영어를 넣도록 학습된 패턴이 강하게 남아있음.

**영향**: 한국어 지식을 저장했어도 **실제로는 영어로 변환되어** 나중에 한국어로 검색할 때 의미 불일치 발생 가능.

---

## 문제 3 — JSON 인수 생성 오류 (한국어 특이 트리거)

**증상**: memory 서버에서 한국어 관찰값을 JSON arguments에 포함하면 파싱 실패.

```
오류:  "Expecting ',' delimiter: line 1 column 261 (char 260)"
원인:  모델이 ~260자 이상의 JSON 생성 시 outer closing } 누락
```

**메커니즘**:
1. 한국어 입력 → 모델이 영어로 확장·번역 → 관찰값이 길어짐
2. JSON 총 길이 260자 초과 → 잘못된 토큰(`']}\n'`) 선택 → `}` 누락
3. `parse_json_blob` 함수가 불완전한 JSON 추출 → 파싱 실패

**7B 비교**: Qwen2.5-7B는 동일 쿼리에서 관찰값을 간결하게 생성 → 임계값 미초과 → 정상 동작.

---

## 문제 4 — 에이전트 오동작·환각 (github 서버)

**증상**: 한국어 쿼리로 커밋 조회를 요청했을 때, 커밋을 찾지 못한 에이전트가 **저장소를 직접 생성**하는 오동작 발생.

```
쿼리:  "rlaehdrjs4609-hub/MCP 저장소의 최근 커밋 3개를 한국어로 요약해줘"
동작:  list_commits(sha='latest') 실패 → "시뮬레이션"이라며 create_repository 호출
결과:  rlaehdrjs4609-hub/rlaehdrjs4609-hub-MCP 저장소가 실제 생성됨
```

**원인**: 모델이 한국어 지시를 "요약"하는 것에만 집중하여, 올바른 tool 선택 대신 "이야기를 완성"하려는 방향으로 오작동.

**영향**: **실제 GitHub 저장소 생성**이라는 되돌리기 어려운 부작용 발생. 한국어 쿼리가 모델의 잘못된 목표 설정을 유발.

---

## 문제 5 — 도구 인수 추론 실패 (filesystem 서버)

**증상**: `read_text_file` 도구에 `tail` 인수가 필수임에도 모델이 이를 찾지 못하고 4회 반복 실패.

```
1회: read_text_file(path='README.md')            → "tail is required"
2회: read_text_file(path='README.md', head=20)   → "tail is required"
3~4회: 동일 반복
```

**대조 실험**: 영어 쿼리도 동일하게 실패. **언어 무관한 3B 모델 능력 한계**.

그러나 한국어 쿼리 환경에서는 **오류 회복 과정에서 영어 전환**(문제 1)이 함께 발생해 체감 품질이 더 낮아짐.

---

## 문제 6 — 컨텍스트 폭증 (github 서버)

**증상**: GitHub API 응답(SHA, URL 등 메타데이터 포함)이 그대로 컨텍스트에 누적되어 처리 지연.

```
한국어 쿼리 github 실험:
  Step 1: 9,454 tokens
  Step 4: 44,842 tokens  (4.7배 증가)
  Step 4 소요: 165초 (CPU) / 8초 (GPU)
```

**한국어 관련성**: 한국어 쿼리 자체가 원인은 아니나, 복잡한 한국어 쿼리일수록 더 많은 API 호출을 유발해 컨텍스트가 빠르게 늘어남.

## 문제별 개선 방향

| 문제 | 즉시 적용 가능한 개선 |
|------|----------------------|
| 언어 전환 | 시스템 프롬프트에 "항상 한국어로 응답하라" 명시 |
| 내용 번역·손실 | 7B 이상 모델 사용 / 한국어 special token 처리 개선 |
| JSON 생성 오류 | `MCPClient(structured_output=False)` 또는 7B 모델 |
| 에이전트 오동작 | 위험 도구(create, delete)에 사전 승인 단계 추가 |
| 도구 인수 추론 | 7B 모델 / tool description에 필수 인수 명시 강화 |
| 컨텍스트 폭증 | API 응답 전처리로 불필요한 메타데이터 제거 |

## 핵심 결론

한국어 쿼리는 MCP 연동에서 다음 두 가지 방식으로 문제를 유발한다:

1. **직접적 영향**: 한국어 텍스트를 tool arguments에 포함할 때 JSON 길이 증가로 생성 오류 트리거 (memory 서버)
2. **간접적 영향**: 한국어 이해·생성 능력이 상대적으로 약한 3B 소형 모델에서, 오류 복구 능력 저하 → 언어 전환, 환각, 오동작으로 이어짐

**3B → 7B 업그레이드만으로 대부분의 문제가 해소**되며, 이는 모델 크기와 한국어 처리 능력이 직결됨을 보여준다.
